# Internationalization (i18n)

Good practice for multilingual frontends emitted by this skill. Read alongside `ui-guidelines/foundations/inclusion.md` (locale-aware defaults), `ui-guidelines/foundations/right-to-left.md` (RTL), and `stack-vanilla-js.md` (the runtime layer).

## What "multilingual" actually means

- **Language** of the strings (English, French, Arabic, …).
- **Locale** of the formatting (dates, numbers, currencies, plurals, sort order).
- **Script direction** (LTR, RTL).
- **Text rendering** — font coverage, typographic conventions.

These four vary independently. A French speaker in Switzerland (`fr-CH`) wants French strings but Swiss number formatting (`1'234,56`). Handle each axis separately.

## Detect the user's language correctly

Order of preference, every time:

1. **Explicit user choice** (a setting in the app, persisted in `localStorage` or a cookie).
2. **URL prefix or subdomain** (`/fr/...`, `fr.example.com`) — this is the single most important signal because it survives sharing.
3. **`Accept-Language` header** server-side or `navigator.languages` client-side.
4. **Hard default** — usually English, sometimes whichever language is the project's primary.

```js
// runtime detection (client side)
const supported = ["en", "fr", "es", "de"];
function pickLanguage() {
  const saved = localStorage.getItem("lang");
  if (saved && supported.includes(saved)) return saved;
  for (const tag of navigator.languages ?? [navigator.language]) {
    const base = tag.toLowerCase().slice(0, 2);
    if (supported.includes(base)) return base;
  }
  return "en";
}
```

## URL strategy

Pick one and apply it everywhere:

| Strategy | Looks like | Use when |
|---|---|---|
| Path prefix | `/fr/about` | Default for marketing + apps. SEO-friendly, easy to share. |
| Subdomain | `fr.example.com` | Languages are run by different teams or need different CDNs. |
| Query string | `/about?lang=fr` | Last resort. Bad for SEO and caching. |
| None | `/about` (content negotiated server-side) | Apps with a single signed-in user; never for marketing. |

Always include `<link rel="alternate" hreflang="…">` tags in the `<head>` so search engines map versions to each other.

```html
<link rel="alternate" hreflang="en" href="https://example.com/en/about">
<link rel="alternate" hreflang="fr" href="https://example.com/fr/about">
<link rel="alternate" hreflang="x-default" href="https://example.com/en/about">
```

## Set `lang` correctly

Every page must declare its language on `<html>`. If a span contains content in another language, mark it on that span.

```html
<html lang="fr">
  …
  <p>The French word for window is <span lang="fr">fenêtre</span>.</p>
</html>
```

For user-generated content fields, use `dir="auto"` so RTL strings render correctly in an LTR page (and vice-versa).

## Strings: separate the message from the code

Never write user-facing strings inline. Keep them keyed in a small dictionary so they can be swapped per language.

```js
// i18n.js
const dict = {
  en: {
    "signin.title": "Sign in",
    "signin.email": "Email",
    "signin.submit": "Continue",
    "delete.confirm": (name) => `Delete ${name}?`,
  },
  fr: {
    "signin.title": "Se connecter",
    "signin.email": "Adresse e-mail",
    "signin.submit": "Continuer",
    "delete.confirm": (name) => `Supprimer ${name} ?`,
  },
};

let current = "en";
export function setLang(l) { current = l; document.documentElement.lang = l; }
export function t(key, ...args) {
  const entry = dict[current]?.[key] ?? dict.en[key] ?? key;
  return typeof entry === "function" ? entry(...args) : entry;
}
```

For larger apps, ship per-language JSON files and lazy-load:

```js
async function loadLang(lang) {
  const res = await fetch(`/i18n/${lang}.json`);
  dict[lang] = await res.json();
  setLang(lang);
}
```

## Formatting — never roll your own

Use `Intl.*`. It ships in every modern browser, handles locale data, and is faster than any hand-rolled formatter.

```js
const lang = document.documentElement.lang || "en";

// numbers
new Intl.NumberFormat(lang).format(1234567.89);            // "1,234,567.89" / "1 234 567,89"

// currency
new Intl.NumberFormat(lang, { style: "currency", currency: "EUR" }).format(99.5);

// dates — short / medium / long / full
new Intl.DateTimeFormat(lang, { dateStyle: "medium" }).format(new Date());

// relative time
new Intl.RelativeTimeFormat(lang, { numeric: "auto" }).format(-2, "day");  // "2 days ago" / "avant-hier"

// list joining
new Intl.ListFormat(lang, { style: "long", type: "conjunction" }).format(["A", "B", "C"]);
```

## Plurals — also `Intl`

English has two forms (singular / plural). French has two but with different rules. Russian has three. Arabic has six. Don't try to encode this with `if (n === 1)`.

```js
const rules = new Intl.PluralRules(lang);
const messages = {
  one:   (n) => `${n} item`,
  other: (n) => `${n} items`,
};
function nItems(n) {
  const form = rules.select(n);            // "one" | "other" | "few" | "many" | …
  return (messages[form] ?? messages.other)(n);
}
```

Per-language plural dictionaries hold the right set of forms; never assume just `one` / `other`.

## Right-to-left

Languages like Arabic, Hebrew, Persian flip layout direction. Use logical CSS properties (`ms-*`, `me-*`, `ps-*`, `pe-*`, `start-*`, `end-*`) — Tailwind supports them.

```html
<html lang="ar" dir="rtl">
```

A handful of things should NOT mirror: brand logos, clocks, media controls, time-series charts that progress left-to-right. See `ui-guidelines/foundations/right-to-left.md` for the full list.

## Fonts and text rendering

- The skill ships Montserrat (Latin + Cyrillic coverage). For scripts not covered (Arabic, CJK, Devanagari, …) load an additional family scoped via `unicode-range`:
  ```css
  @font-face {
    font-family: "Noto Sans Arabic";
    src: url("/fonts/noto-sans-arabic.woff2") format("woff2");
    unicode-range: U+0600-06FF, U+0750-077F, U+FB50-FDFF, U+FE70-FEFF;
    font-display: swap;
  }
  ```
- Use `font-feature-settings` for typographic conventions (French uses spaced punctuation; Spanish uses inverted question marks at the start of questions; some scripts need contextual alternates).
- Use `text-wrap: balance` on headings — works the same in every language.

## Routing & content negotiation

- For static sites, generate one HTML file per language at build time. Don't redirect at runtime — slow and bad for SEO.
- For apps, send the JSON bundle for the matched language and inline the user's language in the initial HTML so there's no flash of wrong-language content.
- Respect HTTP `Accept-Language` only on the first visit; honor the user's persisted choice afterward.

## Pluggable, not bespoke

If the project grows past ~20 strings or needs translator handoff, drop in a real library:

- **i18next** — file-format-agnostic, plurals, interpolation.
- **FormatJS** (`@formatjs/intl`) — ICU MessageFormat.
- **Lit's `@lit/localize`** — for custom-element-heavy apps.

The dictionary pattern above is fine for ≤ 50 strings. Above that, library + translation files in `/locales/{lang}.json`.

## Skill defaults

- Default the runtime language to `navigator.language` matched against the project's supported set.
- Persist the user's explicit choice under `localStorage.lang` and to a cookie for server-side rendering.
- Default to **English** in code and comments; switch the *output language* of the skill (Claude's responses, generated strings) when the user writes in another language.
- For repos that ship in EN + FR, the convention is `README.md` + `LISEZMOI.md` with a `[🇫🇷] · [🇬🇧]` switcher at the top of each file.
- For the alt-text helper: `python front/scripts/alt_from_ollama.py --lang fr image.jpg` to draft alt text in French; defaults to the system `LANG`. See `references/alt-text-ai.md`.

## Checklist

- [ ] `<html lang>` set on every page; `dir="auto"` on user-content fields.
- [ ] One URL strategy across the whole site.
- [ ] `<link rel="alternate" hreflang>` tags in the head, including `x-default`.
- [ ] No inline user-facing strings — every label goes through `t(key)`.
- [ ] Numbers, dates, currencies, relative time, lists, plurals all via `Intl.*`.
- [ ] RTL languages: logical CSS properties throughout; non-mirroring assets identified.
- [ ] Fonts cover the scripts you ship; non-Latin loaded via `unicode-range`.
- [ ] Persisted user choice wins over auto-detection.
- [ ] Alt text drafted in the page's language (set `ALT_LANG` or pass `--lang`).
- [ ] Translator-friendly format if the dictionary exceeds ~50 keys.
